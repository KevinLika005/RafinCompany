2.1 Global header and top-of-page changes
Remove the entire top strip that currently sits above the main website header. This is the strip with the social icons, location, and phone numbers.
Keep the main header/navigation bar, but update it so that the Careers section is also included in the header menu bar.
The Projects item must continue to support the project-category dropdown behavior on hover.
2.2 New Building Materials section
Add a new section for the construction materials used by the company.
The visual reference for this section is the Building Materials section from eberhard.ch.
This is a new section and must be incorporated into the redesigned website structure.
2.3 Careers section requirements
The Careers section must remain part of the website and must also be reachable from the header menu bar.
The Careers section must support filtering. Each job position must have two filters: Career Level and Job Category.
Job Category values must use the categories from the client content file: Transport, Construction, Construction Management, Service and Maintenance, and Vehicle Technology.
Each job position card must follow the supplied reference layout style.
The exact job positions to include are: Truck Driver, Construction Worker, Surveyor, Machinery Mechanic, Electrical Engineer, Mechanical Engineer, Industrial Electrician, Team Leader, and Mechatronics Technician.
2.4 Projects page and project-detail requirements
The projects page filter controls must be redesigned. Instead of the current button set, the page must use a top filter bar styled like the supplied reference image.
This projects filter bar must stay fixed at the top of the page while the user scrolls.
Project previous/next navigation on the single project page must be redesigned to match the supplied reference layout.
The project previous/next navigation must continue to navigate only within the same project group/category, exactly as required in the original client brief.
The existing project-detail behavior from the earlier brief remains in force: project pages open with a hero banner/slider at the top, project information below, and same-category previous/next links before the footer.
2.5 Footer changes
Social media icons in the footer must be larger and more visible.
The footer e-mail field and the Send button must be the same height/size. The current mismatch must be corrected.
The footer must continue to contain the company logo, department contact information, address, and social icons.
2.6 Contacts section
Add a dedicated Contacts section.
This Contacts section must contain a contact data form with a message field.
This Contacts section must also contain two maps for two different locations, with one map on the left side and the other on the right side.

6. Additional recommendations from repository audit (advisory, not locked scope)
Do not redesign only index.html. The top strip, primary navigation, and footer structure are duplicated across index.html, projects.html, category.html, and project.html, so global changes must be applied consistently across all templates.
Create a dedicated materials data source, for example data/materials.js, instead of hardcoding construction materials directly into HTML. This will make the Building Materials section editable without changing templates.
Create a dedicated contacts data source, for example data/contacts.js, to hold the two map locations, addresses, labels, and contact details for the new Contacts section.
Replace placeholder or legacy social links. The current code still contains placeholder Facebook/Twitter/Google Plus/Vimeo/YouTube/Pinterest links in some templates. After the redesign, only the intended channels should remain and all URLs should be real.
Normalize and centralize styling. The current implementation mixes base CSS, patch CSS, page-level inline styles, and style injection from JavaScript. For the redesign phase, Codex should move new section styling into a dedicated stylesheet to reduce drift and duplication.
Keep the current static architecture. The repository already has a workable data-driven layer with /data and /js helpers. A careful extension of this architecture is lower risk than a framework migration.
Preserve and improve the current invalid-slug / empty-state behavior for projects and categories. The code already has fallback handling; it should remain after the redesign and be visually polished.
Add keyboard, hover, focus, and mobile interaction coverage for dropdowns, sticky filters, career filters, project cards, and map embeds.
Because the top strip is being removed, re-check sticky header offsets and scroll calculations on desktop and mobile. The current RD Navbar offsets were tuned for a taller header stack.
Repository findings that justify the recommendations above
Current jobs.js and jobs-section.js only render simple title/category cards, so the careers redesign will require a schema extension and a new renderer.
Current projects.html uses simple button filters rather than a sticky bar, so the filter DOM and CSS both need replacement.
Current project.html uses small primary buttons for previous/next plus a middle all-projects button, so the page layout must be rebuilt to match the supplied navigation reference.
Current data/projects.js contains only a small sample project set and not all categories are populated with real project content, so content migration is required before final QA.
Current footer-company.js already generates footer markup, which is useful, but the design still needs resizing, stronger social icon visibility, and the corrected input/button proportions requested by the client.
